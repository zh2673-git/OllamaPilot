# Opencode 命令自动补全功能分析

## 第一步：点出组件概念

首先，让我们认识构成这个系统的核心"积木块"：

- **[补全对话框组件](internal/tui/components/dialog/complete.go)**：负责显示补全选项列表的对话框组件，就像一个"弹出式菜单"
- **[补全提供者接口](internal/tui/components/dialog/complete.go:62-66)**：定义了获取补全项的"供应商"，负责提供具体的补全内容
- **[文件/文件夹补全提供者](internal/completions/files-folders.go)**：专门提供文件和文件夹补全的"供货商"
- **[聊天页面](internal/tui/page/chat.go)**：主聊天界面，负责监听按键事件并触发补全对话框
- **[命令对话框](internal/tui/components/dialog/commands.go)**：通过快捷键触发的命令选择对话框
- **[模型选择对话框](internal/tui/components/dialog/models.go)**：用于切换AI模型的对话框

**联系说明**：就像一个多功能遥控器，聊天页面（chat.go）监听用户的按键操作。当用户按下不同的快捷键时，会触发不同的"功能"：
- 按 `@` 键就像按下"搜索键"，启动补全系统
- 按 `ctrl+k` 就像按下"菜单键"，打开命令列表
- 按 `ctrl+o` 就像按下"切换键"，打开模型选择

## 第三步：详细解释

### 一、补全对话框的实现原理

#### 1. 补全对话框核心组件

在 `internal/tui/components/dialog/complete.go` 中定义了补全对话框的核心结构：

```go
// 第62-66行：补全提供者接口
type CompletionProvider interface {
    GetId() string
    GetEntry() CompletionItemI
    GetChildEntries(query string) ([]CompletionItemI, error)
}
```

这个接口就像一个"供应商契约"，定义了补全提供者必须实现的三个方法：
- `GetId()`：获取供应商的唯一标识
- `GetEntry()`：获取默认的补全项
- `GetChildEntries(query string)`：**最核心的方法**，根据用户输入的查询字符串返回匹配的补全项列表

#### 2. 补全对话框的显示逻辑

在 `complete.go` 的 `Update` 方法中（第148-152行）：

```go
var query string
query = c.pseudoSearchTextArea.Value()
if query != "" {
    query = query[1:]  // 去掉前缀字符（@或/）
}
```

这段代码的意图是：当用户输入 `@foo` 时，系统会去掉 `@` 前缀，然后用 `foo` 作为查询条件去匹配文件。这说明系统**设计时考虑支持 `/` 前缀**，但当前实现只绑定了 `@` 键。

#### 3. 聊天页面的按键绑定

在 `internal/tui/page/chat.go` 的第39-42行：

```go
ShowCompletionDialog: key.NewBinding(
    key.WithKeys("@"),
    key.WithHelp("@", "Complete"),
),
```

这里只绑定了 `@` 键来触发补全对话框。**没有绑定 `/` 键**，这就是为什么 `/m` 不会自动提示 `/models` 的原因。

### 二、文件/文件夹补全的实现

在 `internal/completions/files-folders.go` 中：

```go
// 第169-185行：获取匹配的文件/文件夹
func (cg *filesAndFoldersContextGroup) GetChildEntries(query string) ([]dialog.CompletionItemI, error) {
    matches, err := cg.getFiles(query)
    if err != nil {
        return nil, err
    }

    items := make([]dialog.CompletionItemI, 0, len(matches))
    for _, file := range matches {
        item := dialog.NewCompletionItem(dialog.CompletionItem{
            Title: file,
            Value: file,
        })
        items = append(items, item)
    }

    return items, nil
}
```

这个函数使用模糊匹配（fuzzy matching）算法来找出与查询条件匹配的文件和文件夹。

### 三、命令对话框和模型对话框

#### 1. 命令对话框（ctrl+k）

- 触发快捷键：`ctrl+k`（定义在 `tui.go:64-67`）
- 功能：显示已加载的自定义命令列表
- 命令来源：从 `~/.config/opencode/commands/`、`~/.opencode/commands/` 和项目目录加载

#### 2. 模型选择对话框（ctrl+o）

- 触发快捷键：`ctrl+o`（定义在 `tui.go:72-75`）
- 功能：让用户选择不同的AI模型

### 四、为什么 `/m` 不会自动提示 `/models`

根据代码分析，**当前版本的 opencode 尚未实现以 `/` 开头的命令自动补全功能**。系统目前只支持：

1. `@` 键触发文件/文件夹补全
2. `ctrl+k` 手动触发命令列表
3. `ctrl+o` 手动触发模型选择

如果需要实现 `/` 命令的自动补全功能，需要：

1. 在 `chat.go` 的按键绑定中添加 `/` 键的监听
2. 创建一个专门处理命令补全的提供者（类似于 `filesAndFoldersContextGroup`）
3. 修改 `complete.go` 中的查询处理逻辑，支持 `/` 前缀

## 总结

opencode 项目的自动补全系统采用了"提供者-消费者"模式，通过定义清晰的接口来支持不同类型的补全功能。目前系统只实现了 `@` 键触发的文件/文件夹补全，而用户期望的 `/` 命令自动补全功能尚未实现，是一个待开发的功能点。
